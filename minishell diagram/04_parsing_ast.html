LEVEL 4 - MEDIUM

🌳 Parsing: Building the Command Tree

Turning flat tokens into a structure the executor can walk through

Why Do We Need a Parser?

The lexer gave us tokens, but tokens are just a flat list. The parser needs to understand the relationships between them: which words belong to which command? Where does the pipe split things? Which file does the redirection point to?

📖 Sentence Analogy: "The cat sat on the mat" — a list of words (tokens). But the PARSER understands: "The cat" = subject, "sat" = verb, "on the mat" = where. It builds MEANING from flat data. Your parser does the same: [ls] [-la] [|] [grep] [.c] → "run ls with -la, pipe its output to grep with .c".

Two Approaches: Linked List vs Tree

At 42, most students choose one of these. Both work — pick what makes sense to you:

Option A: Linked List of Commands

Simpler to implement. Each node = one simple command. Nodes connected by pipe relationship.

✓ Easier to build and debug

✗ Harder to extend later

Option B: Binary Tree (AST)

More elegant. Pipes become tree nodes with left/right children. Closer to how real shells work.

✓ Clean recursive execution

✗ Slightly harder to build

Option A: Command Table (Linked List)

typedef struct s_cmd { char **args; // ["ls", "-la", NULL] (NULL-terminated!) t_redir *redirections; // linked list of redirections struct s_cmd *next; // next command (after pipe) } t_cmd; typedef struct s_redir { int type; // REDIR_IN, REDIR_OUT, APPEND, HEREDOC char *file; // filename or heredoc delimiter struct s_redir *next; // multiple redirections possible! } t_redir;

Option B: Binary Tree (AST)

typedef enum e_node_type { NODE_CMD, // leaf: a simple command with args NODE_PIPE, // internal: left | right } t_node_type; typedef struct s_node { t_node_type type; // For CMD nodes: char **args; t_redir *redirections; // For PIPE nodes: struct s_node *left; struct s_node *right; } t_node;
