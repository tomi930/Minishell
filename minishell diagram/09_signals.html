📡 Signal Handling

How Ctrl+C, Ctrl+D, and Ctrl+\ work (and why they're different)

What Are Signals?

Signals are software interrupts — messages from the OS to a process. They're like someone tapping your shoulder while you're working. You can ignore the tap, do something specific when tapped, or let the default behavior happen (like stopping).

📱 Phone Analogy: Signals are like phone notifications. SIGINT (Ctrl+C) is like someone calling you — you can answer, ignore, or let it go to voicemail (default). SIGQUIT (Ctrl+\) is like a fire alarm — much harder to ignore. Ctrl+D isn't a signal at all — it's like hanging up the phone (closing the input stream).

The 3 Key Inputs in Minishell

Signal Behavior in Different Contexts

⚠ Common Mistake: Not restoring signal handlers. When the parent ignores SIGINT while waiting for a child, it MUST set it back to the custom handler once the child exits and we return to the prompt!

The Code Pattern

// Global variable for signal communication (only ONE allowed!) volatile sig_atomic_t g_signal = 0; void sigint_handler(int sig) { g_signal = sig; write(1, "\n", 1); // newline after ^C rl_on_new_line(); // tell readline we're on new line rl_replace_line("", 0); // clear current input rl_redisplay(); // show fresh prompt } // Setup signals for interactive mode (at prompt): signal(SIGINT, sigint_handler); // Ctrl+C: our handler signal(SIGQUIT, SIG_IGN); // Ctrl+\: ignore // In child process (after fork): signal(SIGINT, SIG_DFL); // restore defaults so child can be killed signal(SIGQUIT, SIG_DFL); // ⚠ Only ONE global variable allowed by 42 subject! // Use it to communicate signal info between handler and main loop
