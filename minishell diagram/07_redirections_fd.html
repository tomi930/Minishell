📂 Redirections & File Descriptors

Understanding the plumbing behind >, >>, <, and <<

First: What ARE File Descriptors?

A file descriptor (fd) is just a number that represents an open file, pipe, socket, or device. Every process starts with 3:

🔌 Plug Analogy: Think of fd 0, 1, 2 as plugs in a wall. Normally all three are plugged into the "terminal" outlet. Redirections unplug one and plug it into a "file" outlet instead. The program doesn't know the difference — it just writes to fd 1 like always. It doesn't care WHERE fd 1 goes.

The 4 Redirections Explained

The Code: How to Do a Redirect

// For: echo "hello" > output.txt // In the CHILD process (after fork, before execve): int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644); if (fd == -1) { perror("minishell: output.txt"); // prints error with filename exit(1); } dup2(fd, STDOUT_FILENO); // fd 1 now points to output.txt! close(fd); // don't need the extra fd anymore // Now execve() the command — its stdout goes to the file! execve("/bin/echo", args, envp);

The beauty of this design: echo has NO IDEA its output goes to a file. It just writes to fd 1 (stdout) like it always does. The shell is the one who changed where fd 1 points.
