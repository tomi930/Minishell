LEVEL 5 - DEEP DIVE

⚡ Execution: fork() + execve()

The moment your shell actually DOES something — the most important syscalls to understand

The Big Idea: Why fork()?

Here's the problem: when you run ls, you need to replace your shell's process with the ls program. But if you do that, your shell is GONE — it can't show the prompt again!

The solution: create a copy of yourself first (fork), then replace the COPY with ls (execve). Your original shell waits safely.

📋 Clone Analogy: Imagine you need to go on a dangerous mission. Instead of going yourself, you create a perfect clone. The clone goes on the mission (runs ls), and you wait at home (waitpid). When the clone is done, you get the report (exit status) and continue your life. If the clone dies... no big deal, you're still here!

fork() — Creating a Clone

🧠 Mind-blowing fact: After fork(), BOTH processes execute the same code. The if (pid == 0) check is how the child and parent take different paths. It's the same code, but pid has a different value in each process!

execve() — Replacing the Clone

Once the child exists, we call execve() to replace it with the actual program (like /bin/ls).
